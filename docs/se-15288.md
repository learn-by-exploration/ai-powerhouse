# Systems Engineering — ISO/IEC/IEEE 15288

This repo integrates the **ISO/IEC/IEEE 15288** system life cycle processes as the spine for how AI Powerhouse helps you build systems end-to-end. ISO 15288 is the international standard for system life cycle processes — it defines 30+ processes across 4 groups, and is the reference most industries use when they say "we do systems engineering."

The standard is intentionally minimal: it specifies *what* each process must achieve, not *how*. That gives us room to map each process to the right Powerhouse tool.

---

## The 4 process groups (overview)

| Group | Count | Purpose |
|---|---|---|
| **Agreement processes** | 2 | Acquire or supply a system (Acquisition, Supply) |
| **Organizational Project-Enabling processes** | 6 | Make the org capable of running projects (Life Cycle Model Management, Infrastructure Management, Portfolio Management, Human Resource Management, Quality Management, Knowledge Management) |
| **Technical Management processes** | 7 | Plan, track, control, and assess the technical effort (Project Planning, Assessment & Control, Decision Management, Risk Management, Configuration Management, Information Management, Measurement, Quality Assurance) |
| **Technical processes** | 11 | The actual engineering work — stakeholder needs → disposal. **This is the spine most engineers interact with.** |

The 11 Technical processes are the lifecycle every system moves through. The other 19 processes exist to make those 11 actually work.

## The 11 Technical processes (the lifecycle)

These are stable across the 2015 and 2023 revisions of 15288. Phases are sequential but in practice you iterate — a verification failure sends you back to requirements, an operation incident triggers maintenance, etc.

### 6.4.1 Stakeholder Needs and Requirements Definition

**Purpose**: Identify the people, organizations, and systems that will interact with the system, and capture what they need from it.

**Outputs**:
- Stakeholder register (who is affected, in what role, with what influence)
- Stakeholder needs statement (prose, one paragraph per stakeholder)
- Operational concept (ConOps) — high-level description of how the system will be used

**Gate to next phase**: Every stakeholder need is traceable to a measurable requirement (or to an explicit "not addressed" decision with rationale).

### 6.4.2 System/Software Requirements Definition

**Purpose**: Translate stakeholder needs into verifiable system requirements.

**Outputs**:
- System requirements specification (functional + non-functional)
- Interface requirements (with other systems, with users)
- Constraints (regulatory, technical, schedule, cost)

**Gate to next phase**: Each requirement is testable, unambiguous, and traced to one or more stakeholder needs.

### 6.4.3 Architecture Definition

**Purpose**: Establish the system's structure — components, their responsibilities, their interfaces, and the design decisions that justify the structure.

**Outputs**:
- Architecture Decision Records (ADRs) — one per significant decision, with context + options + trade-offs + decision + consequences
- Architecture views (C4: Context, Container, Component, Code) or equivalent
- Interface control documents

**Gate to next phase**: Each requirement is allocated to exactly one component, and the architecture supports the non-functional requirements (latency, security, scale, etc.).

### 6.4.4 Implementation

**Purpose**: Build the system components to the architecture and requirements.

**Outputs**:
- Source code, build artifacts
- Unit tests, with the red→green→refactor discipline
- Component-level documentation

**Gate to next phase**: All unit tests pass; code review (at least one approver per change) is complete; static analysis (lint, type check, security scan) is clean.

### 6.4.5 Integration

**Purpose**: Combine components into progressively larger assemblies and verify that they work together.

**Outputs**:
- Integration test plan and report
- Assembled build (incremental: dev → staging → pre-prod)
- Configuration baselines for each integration level

**Gate to next phase**: All integration tests pass; interfaces between components are exercised; the system behaves as a single entity, not as a collection of parts.

### 6.4.6 Verification

**Purpose**: Confirm that the built system satisfies the *system requirements* ("Did we build the system right?").

**Outputs**:
- Verification matrix (requirement → test case → result)
- Test reports at each level (unit, integration, system)
- Defect log (with severity, root cause, and disposition)

**Gate to next phase**: All verification activities complete; no open Critical or High defects; traceability from each requirement to passing evidence exists.

### 6.4.7 Validation

**Purpose**: Confirm that the built system satisfies the *stakeholder needs* ("Did we build the right system?").

**Outputs**:
- Validation scenarios (realistic usage by real users, or simulated)
- Acceptance evidence (per stakeholder need)
- Discrepancy report (where the system falls short of need)

**Gate to next phase**: Each stakeholder need is satisfied in an operational setting, or the gap is accepted in writing by the stakeholder.

### 6.4.8 Transition

**Purpose**: Move the system from development to operations — deploy, install, configure, and train.

**Outputs**:
- Deployment plan (sequence, prerequisites, rollback steps)
- Installation / configuration scripts (Infrastructure as Code)
- Operations handoff (runbooks, on-call rotation, alert routing)
- Training materials for users and operators

**Gate to next phase**: The system is running in its target environment with monitoring active; the operations team has accepted ownership.

### 6.4.9 Operation

**Purpose**: Run the system to deliver its intended service.

**Outputs**:
- Service Level Indicators (SLIs) and Objectives (SLOs)
- Incident response runbook
- Operational dashboards
- Capacity and performance data

**Gate to next phase**: This phase continues until the system is retired. Triggers for moving to Maintenance: defect reports, change requests, or scheduled upgrades.

### 6.4.10 Maintenance

**Purpose**: Sustain the system — fix defects, add capability, adapt to a changing environment.

**Outputs**:
- Change records (what changed, why, who approved, what was tested)
- Updated tests, documentation, and ADRs
- Patch releases / minor version bumps

**Gate to next phase**: Triggers for moving to Disposal: system is obsolete, replaced, or uneconomical to maintain.

### 6.4.11 Disposal

**Purpose**: Retire the system — preserve any data that must be kept, sanitize any data that must not be, and dismantle the operational footprint.

**Outputs**:
- Disposal plan (what gets archived, what gets destroyed, what gets transferred)
- Data sanitization evidence (chain-of-custody, certificates of destruction)
- Post-disposal verification (no live endpoints, no orphaned data, no lingering access)

**Gate to next phase**: The system no longer exists as a live entity; obligations to stakeholders (data retention, audit, support) are fulfilled or formally transferred.

## Mapping to AI Powerhouse tools

This is the table that makes 15288 actionable. Every 15288 Technical process maps to a Powerhouse tool that already implements the process — no duplication, no new skills, just routing.

| 15288 process | Powerhouse tool(s) | Artifact in your repo |

| Stakeholder Needs | `mem-mem-search` (find prior needs) + `master-se-systems-engineer` | `docs/stakeholder-needs.md` |
| Requirements Definition | `superpowers-writing-plans` | Requirements in the plan's spec section |
| Architecture Definition | `ecc-architect` | `docs/adr/0001-*.md`, C4 diagrams |
| Implementation | `superpowers-subagent-driven-development` | Code + tests + build |
| Integration | `ws-agent-teams-team-lead` | Integration test report |
| Verification | `superpowers-test-driven-development`, `ecc-code-reviewer` | Test report, review log |
| Validation | `ws-agent-teams-team-review`, `ecc-verify` | Acceptance evidence per need |
| Transition | `ws-deployment-validation-config-validate` | Deployment plan + rollback runbook |
| Operation | `ws-observability-monitoring-monitor-setup` | Runbook, SLOs, alert routing |
| Maintenance | `ecc-refactor-cleaner`, `superpowers-systematic-debugging` | Change records, defect log |
| Disposal | (no Powerhouse tool — write a `docs/disposal-plan.md` from the Disposal phase prompt) | Disposal plan + sanitization evidence |

The skill `master-se-lifecycle` is the entry point that routes you to the right tool for wherever you are in the lifecycle. The agent `master-se-systems-engineer` is the orchestrator that walks a project through the 11 phases in order, gating each transition on the verification/validation evidence the standard requires.

## How AI Powerhouse itself was built (a 15288 self-audit)

For honesty: AI Powerhouse follows the 15288 lifecycle for its own build, with the following mapping.

| Process | Evidence in this repo |
|---|---|
| Stakeholder Needs | This README, `USAGE.md` "Daily workflows" — what users said they needed |
| Requirements | `submodule-hashes.lock` (reproducibility = integrity requirement), `master/install.sh` flags, the `master-agent-routing` skill's decision tree |
| Architecture | `master/install.sh` (the symlink architecture), `master/CLAUDE.md` (the routing layer) |
| Implementation | 16 submodules, 309 agents, 619 skills, 284 commands — all versioned and hash-pinned |
| Integration | `bash master/install.sh` — the integrator |
| Verification | `bash scripts/verify-submodules.sh`, `.github/workflows/test-install.yml` (CI) |
| Validation | Real-world use by humans invoking the routing skills |
| Transition | This very doc + USAGE.md — the "deployment guide" for the harness |
| Operation | The user runs it; issues go to GitHub Issues |
| Maintenance | `git submodule update --remote --merge` workflow, `scripts/update-hashes.sh` |
| Disposal | Not yet needed; the hash-lock file means we can always pin to a known state |

This self-audit is not a complete SE program — there is no formal Verification matrix, no Validation evidence per stakeholder need, no Disposal plan. Treat this as a starting checklist, not a compliance certificate.
